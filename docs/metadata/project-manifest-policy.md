# Project Manifest Policy

Status: draft
Scope: `my-agents.project.json`

This document defines how to use the optional project-level manifest that bootstraps a repository with the packs, skills, and agents it wants installed into project-scope runtime surfaces.

## Goals

- Make repository bootstrap reproducible across machines.
- Keep project-specific installation choices out of the published package catalog.
- Reuse the same canonical `skills/`, `agents/`, and `packs/` library without inventing another package type.

## Non-Goals

- This document does not define package publishing or release metadata.
- This document does not define environment overlays or per-member version pinning.

## What the Manifest Is

`my-agents.project.json` is a repository consumer configuration file.

It answers:

- which packs should be installed into this repository
- which extra standalone skills should be installed
- which extra standalone agents should be installed
- which platforms this repository expects by default

It is not a publishable package and should not appear in generated library catalogs.

## Recommended Shape

```json
{
  "schemaVersion": 1,
  "description": "Bootstrap manifest for a product-focused repository.",
  "platforms": ["claude", "codex"],
  "packs": ["product-manager"],
  "skills": [],
  "agents": []
}
```

## Membership Guidance

Use the manifest fields as follows:

- `packs`: the preferred path for curated role or team bundles
- `skills`: direct additions that are intentionally outside the selected packs
- `agents`: direct additions that are intentionally outside the selected packs

Prefer packs first. Use direct `skills` and `agents` only to add small local customizations on top of a pack-based baseline.

## Platform Guidance

`platforms` is optional.

Resolution order:

1. CLI `--platform` passed to `npm run sync-project`
2. manifest `platforms`
3. default to all supported platforms

This lets a repository declare its default surfaces while still allowing one-off operator overrides.

## Sync State And Pruning

`sync-project` keeps a local runtime state file at `.my-agents/project-sync-state.json`.

This file is:

- project-local
- generated by the sync tool
- intended to stay out of version control

Why it exists:

- plain `sync-project` installs missing desired items but does not remove previously managed ones
- `sync-project -- --prune` uses the state file to determine which skills and agents were previously managed and are now safe to remove

The tool only prunes members it previously managed. It should not remove unrelated installs that were never recorded in the sync state.

Without `--prune`, the managed state grows or stays stable as new items are added. Use `--prune` when you want the project runtime surfaces to converge back to the manifest's current desired set.

## Commit Guidance

Commit `my-agents.project.json` when the repository wants a shared, reproducible bootstrap contract for collaborators.

Do not commit it if the file only contains personal experimentation or machine-specific preferences. In that case, keep a local copy outside version control or use a different local filename with `--manifest`.

## V1 Limits

For v1:

- no manifest inheritance
- no lockfile
- no per-platform member overrides inside the manifest

Future versions can add those concepts if real usage demands them.
